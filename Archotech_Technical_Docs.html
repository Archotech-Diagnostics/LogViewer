Neural Interface: Connected // Technical Documentation Alpha 1.6

Overview: The Diagnostic Engine Architecture

Archotech Diagnostics operates as a low-level instrumentation framework designed to resolve the "Black Box" nature of RimWorld's modded environment. The engine is built on a foundation of **Dynamic Binary Instrumentation (DBI)**, utilizing the Harmony library to perform runtime bytecode manipulation. Our architecture prioritizes execution speed by adhering to a **Zero-Allocation hot-path**, utilizing Span<T> and stackalloc for string parsing and metadata extraction to bypass the Managed Heap whenever possible.

Engine Hooking Strategy: We utilize **Method Redirection (Detouring)** on critical Unity engine paths. By hijacking Application.logMessageReceived and Log.Message, we create a non-blocking diagnostic stream that intercepts data before it reaches the game's internal Log.messages buffer.

1. Deep Scan (ModMapper & Heuristics)

ModMapper is a highly-optimized file system crawler that operates independently of the game's standard ModContentPack initialization. It is designed to identify "Invisible Performance Debt"—unoptimized assets that consume system resources without ever appearing in a traditional code-based profiler.

VRAM & Memory Estimation Logic

The scanner doesn't just list files; it performs a **Heuristic Memory Projection**. For every .png or .dds detected in /Textures, the engine calculates the estimated Texture2D footprint using the following C# logic:

// Heuristic Memory Calculation long totalBytes = width * height * (hasAlpha ? 4 : 3); if (supportsMipMaps) totalBytes = (long)(totalBytes * 1.33f); // Accounting for MipMap overhead if (textureCompression == DXT1) totalBytes /= 8;

By comparing file sizes on disk with projected memory usage, ModMapper can identify "VRAM Bombs"—textures that are small on disk due to PNG compression but expand to hundreds of megabytes in GPU memory because they lack proper DXTC/BC7 metadata.

Threaded I/O & Recursive Indexing

The indexing engine utilizes System.Threading.Tasks.Parallel.ForEach with a custom MaxDegreeOfParallelism tuned to the user's CPU core count. It performs a recursive walk of the RootDir, identifying symbolic links and junctions that can often cause infinite loops or duplicate scanning in less robust tools. For assemblies, it uses System.Reflection.AssemblyName.GetAssemblyName() to extract metadata without fully loading the DLL into the AppDomain, preventing "Ghost Assembly" conflicts.

2. Debug Log - Info View (Enhanced Analysis)

The Enhanced Debug Log is the mod's primary diagnostic interface. It works by intercepting the UnityEngine.Application.logMessageReceived event and augmenting the incoming data with Archotech-specific metadata.

Recursive Stack Walker & Assembly Mapping

When an error occurs, the Suspect Analyzer performs a reverse-order walk of the StackTrace. For every frame, it extracts the MethodBase and utilizes an internal cache (Dictionary<Assembly, ModContentPack>) to resolve the method's origin. This is significantly faster than standard reflection-based lookups.

  • Harmony Detection: If a method has been patched, we utilize Harmony.GetPatchInfo(method) to retrieve the Prefixes, Postfixes, and Transpilers. We then iterate through the owner IDs of these patches to identify "Accomplices"—mods that are potentially modifying the method's behavior even if they didn't throw the initial exception.
  • IL Offset Correlation: For advanced debugging, we capture the IL offset of the instruction that failed, allowing modders to correlate the crash with specific lines in their source code via the LogViewer's detailed view.

DS Trace & Log Suppression Hashing

The spam filtering system utilizes a High-Performance MurmurHash3 algorithm. Every incoming log message is converted into a 128-bit hash. Identical hashes are stored in a ConcurrentDictionary<Guid, LogEntry>. When a repeat is detected, we simply increment the OccurrenceCount on the existing entry. This prevents the "Log Buffer Overflow" that typically causes frame-rate drops when a mod enters an infinite error loop (e.g., a JobDriver failing every tick).

3. Load Time Diagnostics (Startup Instrumentation)

Load Time Diagnostics is the most complex component of the mod, requiring surgical **Harmony Transpilation** of the game's core loading methods in Verse.ModContentPack and Verse.DirectXmlLoader.

XML Hydration Profiling

Traditional profilers can only tell you that the game is "Loading Defs." Our system transpilers inject Stopwatch start/stop calls directly into the DirectXmlLoader.LoadXmlData<T> loop. This allows us to measure the "Hydration Time" for every individual XmlNode as it is converted into a C# Def object. This is critical for identifying mods with massive, poorly-indexed XML files that cause the "Processing..." hang during startup.

Static Constructor (C#) Tracking

The system patches Verse.StaticConstructorOnStartupUtility.CallAll, wrapping the call to RuntimeHelpers.RunClassConstructor for every class marked with the [StaticConstructorOnStartup] attribute. We track the execution time of these constructors, which is where many mods perform heavy tasks like GraphicDatabase population or complex DefDatabase searches that delay the Main Menu appearance.

4. Performance Scan (Real-Time Profiling)

Performance Scan operates at the simulation level, monitoring the health of the TickManager and the Unity MainLoop.

Tick-Rate Variance & Simulation Spikes

We monitor the execution time of Verse.TickManager.DoSingleTick. By comparing the Stopwatch delta against the expected tick-rate (e.g., 16.6ms for 1x speed), we can identify "Simulation Choke." When a choke is detected, the engine dynamically samples the most active modded methods in the Pawn_JobTracker and Pawn_PathFollower classes, which are the most common sources of TPS drops in late-game colonies.

Unity MainThread Monitoring

To detect UI and rendering stutters, we hook into the UnityEngine.MonoBehaviour.Update and OnGUI cycles. We track the Frame-Time Standard Deviation; if the variance exceeds a specific threshold (micro-stuttering), we capture a snapshot of the current EventStack. This allows us to identify if the stutter was caused by a mod's Draw call or a heavy Harmony postfix on a rendering method.

5. General Export & Infrastructure

The export system is a high-speed serialization pipeline designed to package the entire diagnostic state into a compressed JSON payload.

Polymorphic Serialization & Compression

We utilize a custom JSON serialization engine that handles the game's complex, often circular, object graphs. The data includes full stack traces, mod versioning, and hardware entropy. Before transmission, the payload is compressed using System.IO.Compression.GZipStream to minimize bandwidth and ensure fast uploads even on slow connections.

Cloud Layer: Gists & Workers

The infrastructure is entirely serverless, ensuring maximum stability and zero maintenance.

  • GitHub Gist Backend: Reports are stored as anonymous Gists. This provides a globally distributed, high-availability storage solution with built-in versioning.
  • Cloudflare Worker Layer: A serverless middleware layer acts as the "Traffic Controller." It handles report routing, provides a custom URL-shortening service for easy sharing, and enforces **Rate-Limiting** to prevent abuse of the export system.
  • LogViewer Hydration: The web-based LogViewer is a static Single-Page Application (SPA). It "hydrates" the Gist data on the client side, performing the final rendering of the mod list and logs entirely in the user's browser, ensuring that sensitive diagnostic data is only ever processed locally.
